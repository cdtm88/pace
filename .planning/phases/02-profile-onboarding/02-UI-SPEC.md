## Spacing Scale

Declared values (must be multiples of 4). Inherited from Phase 1 8-point scale:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline spacing between label and input |
| sm | 8px | Between field label and input (`space-y-2`) |
| md | 16px | Between form fields (`space-y-4`), step indicator gap |
| lg | 24px | Between field groups (`space-y-6`) |
| xl | 32px | Card internal padding (`p-8 = 32px`) |
| 2xl | 48px | Page vertical centering buffer |
| 3xl | 64px | Unused in this phase |

**Exceptions:**
- Input height: 48px (`h-12`) — iOS touch target minimum; also prevents iOS Safari auto-zoom (PWA-05 prerequisite)
- Submit button height: 48px (`h-12 w-full`) — matches Phase 1 auth button pattern
- Progress bar: 8px height (`h-2`) — thin indicator, not a structural element
- Wizard container: max-w-lg (512px) — wider than auth max-w-sm to give textarea fields room to breathe

**Source:** Phase 1 `login-form.tsx` establishes h-12 inputs and space-y-6 field groups. Carry forward exactly.

## Typography

Exact values from Phase 1 implementation — do not deviate:

| Role | Size | Weight | Line Height | Tailwind |
|------|------|--------|-------------|----------|
| Body | 14px (text-sm) | 400 regular | 1.5 | `text-sm` |
| Label | 16px (text-base) | 500 medium | 1.5 | `text-base font-medium` |
| Heading | 24px (text-2xl) | 600 semibold | 1.2 | `text-2xl font-semibold leading-[1.2]` |
| Display | 28px | 700 bold | 1.1 | `text-[28px] font-bold leading-[1.1] tracking-tight` |

**Sizes used in this phase: 14, 16, 24, 28** (4 sizes, matching Phase 1)
**Weights used: 400, 500, 600, 700** — but only 2 semantic weights: regular (body/helper) and semibold+ (labels/headings/display)

**Phase 2 additions:**
- Textarea body text: 16px (`text-base`) at weight 400 — must be ≥16px to prevent iOS auto-zoom on focus (PWA-05)
- Step label (e.g. "Step 1 of 3"): 14px (`text-sm`) at weight 400, `text-muted-foreground`
- Optional field hint (e.g. "Optional"): 14px (`text-sm`) at weight 400, `text-muted-foreground`
- Dashboard welcome heading: 24px (`text-2xl font-semibold`) — consistent with auth heading role
- Zone badge (Coggan Z1–Z7, RPE label): 14px (`text-sm`) at weight 500, `text-muted-foreground`

**Source:** `login-form.tsx` — `text-2xl font-semibold leading-[1.2]` for heading; `text-base font-medium` for labels; `text-sm` for muted/helper. `auth-card.tsx` — `text-[28px] font-bold leading-[1.1]` for wordmark.

**Accent (`--primary` white) reserved for:**
1. Primary submit button background on each wizard step ("Next" / "Get started")
2. "Save profile" button on profile edit page
3. "Edit profile" link/button on dashboard (if rendered as a filled button)

**Accent is NOT used for:** step indicator fill, progress bar fill, badges, links, hover states, muted text, helper text, zone labels.

**Progress bar fill:** Use `--secondary` (`#27272a` zinc-800) for progress track; use `--primary` white for filled portion — this keeps the progress indicator high-contrast without introducing a new color.

**Dashboard stub surface:** Same `--card` zinc-900 as wizard card.

**Source:** `globals.css` Phase 1 comment block + `.dark {}` token definitions. All color decisions locked by Phase 1.

## Screen Contracts

### Screen 1: Onboarding Wizard (`/onboarding`)

**Layout:** Full viewport zinc-950 background. Vertically centered single column. Wordmark above card (same as auth screens). Card: `max-w-lg w-full bg-card p-8 rounded-xl`. Step indicator below wordmark, above card heading.

**Structure (top to bottom):**
1. Wordmark — "Pace" — 28px/700 — `text-foreground` — same as auth screens
2. Step progress bar — `Progress` component — `h-2 w-full max-w-lg` — shows completion fraction (1/3, 2/3, 3/3)
3. Step counter label — "Step 1 of 3" — 14px/400 — `text-muted-foreground text-right` — above card
4. `Card` / `CardContent p-8` container
5. Step heading — 24px/600 — `text-foreground leading-[1.2]`
6. Step subheading / helper — 14px/400 — `text-muted-foreground` — one line only
7. Field(s) with label + input/textarea
8. Error banner (if present) — above buttons
9. Button row — Back (ghost) left, Next/Submit (default/white) right OR full-width Submit on Step 3

**Step 1 — Training Goals:**
- Heading: "What are your training goals?"
- Helper: "Tell Claude what you're working toward. Be specific — more detail means better sessions."
- Field: `Textarea` — label "Goals" — `min-h-[120px]` — `placeholder="e.g. Build base fitness for a gran fondo in October, currently riding 3x week at Z2"`
- Required: yes — show inline error if submitted blank
- Buttons: "Next →" (default, right-aligned or full-width)
- No Back button on Step 1

**Step 2 — Injury & Health Notes:**
- Heading: "Any injuries or health considerations?"
- Helper: "Optional. Claude uses this to avoid exercises that could aggravate existing issues."
- Field: `Textarea` — label "Injury & health notes" — `min-h-[100px]` — `placeholder="e.g. Left knee tendinitis — no high-cadence sprints; cleared for threshold work"`
- Optional badge: render "(Optional)" in label as `text-muted-foreground text-sm font-normal` span after label text
- Required: no — can submit blank
- Buttons: "← Back" (ghost, left) + "Next →" (default, right)

**Step 3 — FTP & Weight:**
- Heading: "Your training data"
- Helper: "Both are optional. FTP unlocks power zone labels and TSS estimates. You can add these later."
- Field A: `Input` — label "FTP" + "(Optional)" — `type="number" inputMode="numeric" min="50" max="700"` — `placeholder="e.g. 250"` — unit hint "watts" as `text-sm text-muted-foreground` after input
- Field B: `Input` — label "Weight" + "(Optional)" — `type="number" inputMode="decimal" min="30" max="250"` — `placeholder="e.g. 70"` — unit hint "kg" as `text-sm text-muted-foreground` after input
- Required: neither field required
- Buttons: "← Back" (ghost, left) + "Get started" (default, full-width or right)
- Submit triggers `saveProfileAction` Server Action

**Loading state (Step 3 submit):** Button shows `<Loader2 className="animate-spin" />` + "Setting up your profile…" — button disabled + `aria-busy="true"` — same pattern as auth forms.

**Transition between steps:** No animation in v1. Replace step content synchronously on "Next" click. No page navigation.

### Screen 3: Profile Edit (`/profile`)

**Layout:** Same as dashboard — zinc-950 background, single column, max-w-lg, `pt-16`. Wrapped in a `Card` with `p-8` padding (same as wizard card).

**Structure (top to bottom):**
1. Back link — "← Dashboard" — `text-sm font-medium text-foreground` — above card
2. Card with `CardContent p-8`
3. Page heading — "Edit profile" — 24px/600 — `text-foreground leading-[1.2]`
4. Subtext — "Changes are saved immediately." — 14px/400 — `text-muted-foreground`
5. All four fields (Goals textarea, Injuries textarea, FTP input, Weight input) — same specs as wizard but on one page, pre-populated via RSC
6. Error banner (if present) — above submit
7. Submit button — "Save profile" (default/white, `h-12 w-full`)
8. Loading state: "Saving…" + Loader2 spinner (same pattern as auth/wizard)

**Pre-population:** Fields default to existing profile data fetched via RSC. Empty optional fields show placeholder text. Required Goals field shows existing value.

**On save success:** Server Action calls `revalidatePath('/dashboard')` then `redirect('/dashboard')`. Profile page redirects back to dashboard.

## Interaction Contracts

### Wizard Step Navigation

| Action | Trigger | Behavior |
|--------|---------|---------|
| Next (Step 1→2) | Button click | Validate Goals not blank; if blank show inline error "Describe your training goals to continue."; if valid update step state to 2 |
| Next (Step 2→3) | Button click | No validation (optional field); update step state to 3 |
| Back (Step 2→1) | Button click | Update step state to 1; preserve entered text in formData state |
| Back (Step 3→2) | Button click | Update step state to 2; preserve FTP/weight in formData state |
| Submit (Step 3) | Button click / form submit | Trigger Server Action; show loading state; on error show ErrorBanner; on success redirect to /dashboard |
| Tab navigation | Tab key | Moves through label → input → button in DOM order |
| Skip FTP/Weight | Leave inputs blank, click Submit | Server Action receives undefined for ftp/weight; saves null to DB; valid |

### Form Inputs

| Input | Type | inputMode | autoComplete | Min Height |
|-------|------|-----------|--------------|------------|
| Goals textarea | textarea | (default) | off | 120px |
| Injuries textarea | textarea | (default) | off | 100px |
| FTP input | number | numeric | off | 48px (h-12) |
| Weight input | number | decimal | off | 48px (h-12) |

**iOS auto-zoom prevention:** All inputs and textareas use `text-base` (16px font-size). This is a hard requirement per PWA-05.

### Error Handling

| Error | Display | Copy |
|-------|---------|------|
| Goals field blank on Step 1 submit | Inline below field — `text-sm text-destructive` + `aria-invalid` on textarea | "Describe your training goals to continue." |
| FTP out of range (50–700W) | Inline below FTP field | "FTP must be between 50 and 700 watts." |
| Weight out of range (30–250kg) | Inline below Weight field | "Enter your weight in kilograms (30–250)." |
| Server Action failure (DB error) | `ErrorBanner` above submit button | "Something went wrong. Please try again." |
| Session expired mid-wizard | Server Action redirects to /login | No client-side message — redirect is the signal |

### Accessibility Contract

- All form fields have `<Label htmlFor>` with matching `id`
- Error messages linked via `aria-describedby` pointing to error element `id`
- Invalid fields carry `aria-invalid="true"`
- `ErrorBanner` carries `role="alert"` (already implemented in Phase 1)
- Submit buttons carry `aria-busy="true"` + `disabled` during pending state
- Progress bar uses native `<progress>` or shadcn Progress with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`
- Step heading is `<h1>` — one per page
- "Back" and "Next" buttons are `type="button"` to prevent accidental form submit
