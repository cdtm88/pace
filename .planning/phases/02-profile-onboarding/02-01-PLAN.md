## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| migration → production DB | DDL applied to a table with existing Phase 1 rows |
| caller → query helper | userId passed to findUserProfileByUserId is the authoritative session id (set by callers in later plans) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering (IDOR) | findUserProfileByUserId | mitigate | Single `eq(userProfiles.userId, userId)` returns only the caller's own row; userId is the resource identifier (one profile per user). No secondary resource id, so the and() two-condition rule does not apply. |
| T-02-02 | Denial of Service | onboarding_complete migration on existing rows | mitigate | `.notNull().default(false)` guarantees existing Phase 1 rows satisfy the constraint (Pitfall 4); migration is non-destructive. |
| T-02-03 | Tampering | onConflictDoUpdate target | mitigate | `.unique()` on userId is the constraint the upsert (Plan 02) targets; without it the upsert throws (Pitfall 3). Added in this plan as the prerequisite. |
| T-02-SC | Tampering | npm installs | mitigate | No new npm packages this plan (RESEARCH Package Legitimacy Audit: zero installs). shadcn component installs are in Plan 02. |
</threat_model>
