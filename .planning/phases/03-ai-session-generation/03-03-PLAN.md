## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Server Action | The component sends only `readinessScore` (0–3); all auth, profile, validation, and persistence happen server-side in generateSessionAction |
| server action result → UI | Only `{ data?, error? }` reaches the client; `rawJson` and technical details stay on the server |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Information Disclosure | Technical detail / raw_json leaking into the UI | mitigate | Component renders only `result.data` summary fields (title/duration/blocks) and the generic `error` string; never `rawJson` or Zod issue text (acceptance criterion greps rawJson count == 0) |
| T-03-08 | Tampering | Client supplying an out-of-range readinessScore | accept | The score is a UI-bounded 0–3 selector; the server prompt treats it as a label lookup and the AI output is still gated by Zod + safety regardless of readiness value — no DB trust placed in the client number beyond storage |
| T-03-06 | Information Disclosure | ANTHROPIC_API_KEY reaching the client | mitigate | The component never imports the key or the SDK; it only imports the `'use server'` action, which runs server-side (Phase 1 D-13) |
</threat_model>
