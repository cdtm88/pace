<objective>
Implement the three auth route handlers (signup, login, logout) and the rate-limiting module. Signup enforces SIGNUP_ENABLED with the first-user bootstrap bypass (D-11) and hashes passwords with bcryptjs CF12 (D-05). Login applies dual-axis Upstash rate limiting (D-10) and returns a generic anti-enumeration error (D-07). Logout destroys the session and redirects. All mutations run in Route Handlers (not RSC) per the Architectural Responsibility Map — session.save()/destroy() cannot run in RSC.

Purpose: This delivers the core auth capability — users can sign up, log in across browser sessions via httpOnly cookie, and log out. It implements AUTH-01 through AUTH-05.
Output: Working /api/auth/{signup,login,logout} endpoints and src/lib/ratelimit.ts; auth.test.ts and ratelimit.test.ts turn green.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /api/auth/* | Untrusted email/password POST bodies cross into route handlers |
| route handler → Neon | User lookups/inserts cross into the DB |
| route handler → Upstash | Rate-limit counters cross into Redis over HTTP |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-02 | Spoofing | signup/route.ts | mitigate | bcryptjs hash CF12 (D-05); first-user bypass restricted to empty table; unique email constraint blocks duplicate accounts (Pitfall 5). |
| T-1-03 | Information Disclosure | login/route.ts | mitigate | Identical "Invalid email or password" for wrong email vs wrong password (D-07); bcryptjs.compare is timing-safe. |
| T-1-05 | Elevation of Privilege | login rate limiting | mitigate | Dual-axis Upstash limiter — per-IP 10/15m AND per-email 5/15m; block on either (D-10, Pitfall 6) defeats credential stuffing from one IP or distributed enumeration of one account. |
| T-1-04 | Spoofing | session cookie | mitigate | iron-session httpOnly + secure + sameSite=lax, 30-day maxAge (D-06); signed/tamper-proof cookie; payload { id, email } only (D-04). |
| T-1-CSRF | Tampering | all auth POST routes | accept | SameSite=Lax sufficient for same-origin form POSTs (D-12); no separate CSRF token library in v1. |
| T-1-INPUT | Tampering | route handler bodies | mitigate | Zod v4 schemas validate every body (V5); z.email() + min(8); reject before DB/hash. |
</threat_model>
